The Bot That Sees Through Every Mask

In the anarchic wilds of 8b8t.me — a server where the rules are few, the players many, and the cosmetic nick plugin renders identity a matter of pure theatre — there exists one entity that cannot be fooled. DPS_Gemini does not care what your nick says. It knows who you are. It has always known.

The secret is not magic, nor is it surveillance. It is engineering. When the Dead Poets Society's resident AI bot receives a chat message, it does not read the name the server displays. It reaches into the raw binary of the Minecraft packet protocol itself, extracts the clickEvent that the server's cosmetic plugin helpfully signs with your real profile name, strips the /msg  prefix, and there you are. Caught. Identified. Welcomed.

This is the story of that bot: how it was built, why it matters, what it does, and why Dead Poets Society's members — and the broader 8b8t community — owe a small but sincere debt of gratitude to Freddison, its architect, and to the Gemini 2.5 Flash model, its beating heart.

Origins & History

How DPS_Gemini Came to Be

From a Simple Idea to an Engineering Problem

Every great piece of software begins with a frustration. For Freddison, that frustration was simple: 8b8t's cosmetic nick plugin made it impossible for a bot to reliably identify who was speaking. Players could set their display name to anything — to an impersonation, to a joke, to the name of another clan member entirely — and bots built on mineflayer's default event system would be fooled every single time.

The first generation of approaches was reasonable. Cross-reference the speaking player's displayed name against bot.players — the internal mineflayer player list. Surely the list would hold real names? It did not. The cosmetic nick plugin, being thorough in its deception, updates the tab list entry alongside the chat format. Both display the nick. The real name is nowhere on the surface.

The breakthrough came from reading the plugin's source code. In ChatListener.java, the chat formatter appends a clickEvent to every player's name in chat — a "click to message" feature. This clickEvent uses player.getName(), not the display name. It is a direct leak of the real profile username, embedded in every single public chat message, signed by the server itself.

The solution was to go below mineflayer entirely. Rather than relying on any parsed event, DPS_Gemini hooks bot._client.on('packet') — the raw node-minecraft-protocol layer — and reads the JSON component tree before mineflayer touches it. The real username is always there, waiting to be extracted.

Technical Context

The cosmetic nick plugin at 8b8t.me uses the Kyori Adventure library to build complex Component objects for display names. These components are then applied via player.displayName(component) and player.playerListName(component).

The fatal error — from the nick's perspective — occurs in the chat formatter, where ClickEvent.suggestCommand("/msg " + player.getName() + ' ') is called. The real name is embedded at serialization time.

DPS_Gemini exploits this by recursively walking the deserialized JSON component tree via findClickEventValue(), locating the first node where clickEvent.action === 'suggest_command' and value.startsWith('/msg ').

Technology — Under the Hood

The Architecture of Certainty

The Stack

Runtime: Node.js with mineflayer

AI Model: Gemini 2.5 Flash via @google/genai SDK

Packet Layer: node-minecraft-protocol (via bot._client)

Target Server: 8b8t.me (1.20.1, offline auth)

Thinking Mode: ThinkingLevel.NONE

Max Tokens: 1,000 output tokens per call

History Window: Last 4 turns (≈2 exchanges)

Auth: Approved players list via flat text file, loaded at startup

Gemini 2.5 Flash: Speed Without Sacrifice

The choice of Gemini 2.5 Flash was deliberate and, frankly, perfect. In a Minecraft chat context, responses need to arrive within seconds. A model that pauses to "think" — to run extended chain-of-thought reasoning across hundreds of tokens — is the wrong tool entirely. DPS_Gemini uses ThinkingLevel.NONE, disabling the model's internal reasoning pass and going straight to response generation.

The result is a bot that feels immediate. You ask it something. It answers. The response is capped at two hundred characters — not because the model can't do more, but because this is Minecraft chat, and nobody wants to receive a dissertation in seventeen separate messages about whether a ghast can be killed by its own fireball.

Conversation history is maintained across the last four message turns per user, stored in a Map<string, Message[]>. This means the bot remembers context within a session — ask it a follow-up and it knows what you're following up on. Older turns are pruned automatically to keep the prompt window manageable.

A global API cooldown of five seconds between calls prevents rate-limit hammering. Per-user quotas of five messages per two-minute window ensure no single player can monopolise the bot's attention. Both systems work in concert, with graceful countdown messages sent when limits are hit.

How the Trigger Detection Works

The trigger regex /(?:^|>)\s*!g(?:emini)?,?\b/i matches any of: !g, !gemini, >!g, > !g, >!gemini, > !gemini — and all of these with a trailing comma (for when autocorrect strikes). The trigger can appear anywhere in the first three words of a message. Whispers bypass trigger detection entirely: if you whisper the bot directly, your entire message becomes the prompt.

Once a trigger is confirmed, stripTrigger() removes it cleanly, and the remaining text is forwarded as the prompt to handleGeminiRequest() with alreadyStripped=true, bypassing the redundant check in the handler itself. Clean, fast, unambiguous.

01
Identity Resolution

The centrepiece. Every public chat message on 8b8t carries a JSON component with a clickEvent whose value is /msg [real_name] . DPS_Gemini walks the component tree recursively — checking extra[], with[], and nested objects — until it finds this value, then strips the prefix. The result is the player's actual Minecraft profile name, regardless of what cosmetic nick they have set.

02
Resilient Reconnection

Servers kick bots. It's a fact of life on 8b8t — anti-AFK measures, maintenance, the occasional chaos. DPS_Gemini responds to every end and kicked event by scheduling a reconnection fifteen seconds later. It will attempt this up to ten thousand times before giving up. In practice, ten thousand reconnect attempts at fifteen-second intervals represents over 41 days of continuous retry. The bot does not give up easily.

03
Message Architecture

Minecraft's chat system has a hard 256-character limit per message. DPS_Gemini's sendSmartChat() function respects this by first cleaning the response (removing newlines, extra spaces, markdown symbols), then either sending it as a single /msg command if it fits, or splitting it at sentence boundaries — and then word boundaries — into chunks, sent with a two-second delay between each.

Development History

A Chronicle of Engineering Decisions

Generation I

The Naïve Approach: Trusting the Chat Event

The first implementation used mineflayer's built-in chat event, which parses a username field directly. On vanilla servers, this works perfectly. On 8b8t with the cosmetic nick plugin, it occasionally returns the player's nick instead of their real name, depending on how the server formats the chat component. Unreliable. Abandoned.

Generation II

The Cross-Reference Approach: Trusting the Player List

The second iteration added a getRealUsernameMap() function that pulled from bot.players — mineflayer's internal map of connected players. The theory: mineflayer builds this from Player Info packets, which contain real profile names. The practice: the cosmetic nick plugin updates the tab list display name to match the nick, so the player list key was the nick, not the real name. A player nicknamed "DPS_Chatbridge" would appear in the list as "DPS_Chatbridge." Identity still unresolvable.

Generation III

The Packet Intercept: Reading the Source

The current and final approach abandons all mineflayer abstractions and hooks directly into bot._client.on('packet'). The raw JSON component payload is parsed, the component tree is walked recursively via findClickEventValue(), and the /msg realname  value is extracted from the server's own clickEvent signature. This works because the server itself cannot avoid using the real name here — it is used for the actual messaging command routing.

Current

Trigger-First Architecture & Clean Logging

The latest refinement moved trigger detection to the first step: the raw packet is parsed for a trigger word before any API call is considered. This means the bot processes only the messages it needs to, ignoring everything else at minimal computational cost. The logging was stripped to essentials — [Chat], [Whisper], [Request], [Response], [Blocked]. Clean console. Clean code. Clean conscience.

Technical Analysis — For the Curious

The Code That Makes It Work

The Heart of the Matter: findClickEventValue()

The most important function in the entire codebase is fewer than twenty lines. It walks a Minecraft JSON chat component tree — a potentially deeply-nested structure of objects with extra[] and with[] arrays — and returns the first suggest_command clickEvent whose value begins with /msg.

The recursion handles arbitrary nesting depth. A chat component might have a name node nested three levels into the extra array of a translation component — the function does not care. It will find it.

// Recursively finds the real /msg clickEvent value
function findClickEventValue(component) {
  if (!component || typeof component !== 'object') return null;

  if (component.clickEvent?.action === 'suggest_command') {
    const val = component.clickEvent.value || '';
    if (val.startsWith('/msg ')) return val;
  }

  for (const child of (component.extra || [])) {
    const found = findClickEventValue(child);
    if (found) return found;
  }
  for (const child of (component.with || [])) {
    const found = findClickEventValue(child);
    if (found) return found;
  }
  return null;
}

// Extract real username: "/msg Alice " → "Alice"
function realUsernameFromClickValue(value) {
  return value.replace(/^\/msg\s+/, '').trim();
}
Why This Works

The cosmetic nick plugin's source code in ChatListener.java contains this line:

.clickEvent(ClickEvent .suggestCommand("/msg " + player.getName() + ' '))

Note: player.getName() — not player.displayName(). The real name is hardcoded into the packet at the Java level, before serialization. No client-side trick can change this. The server signs it. We read it.

This is the clickEvent "leak" identified in the plugin architecture review. DPS_Gemini's entire identity resolution system is built on this single, reliable, server-authoritative fact.

Frequently Asked Questions

Common Queries, Answered Definitively

Can I trick DPS_Gemini with a nick?

No. The bot reads the server's own clickEvent signature, which uses your real profile name. Your nick is irrelevant. You could call yourself "NotAPlayer" and the bot would still know you are, in fact, a player.

What triggers the bot?

Any public chat message containing !g, !gemini, >!g, > !gemini, or any of those with a trailing comma. Whispers require no trigger — your entire message becomes the prompt.

What happens when the server kicks the bot?

It waits fifteen seconds and reconnects. Then again. Then again. Up to ten thousand times. This is not stubbornness — it is engineering resilience made manifest.

Why does it respond so quickly?

Gemini 2.5 Flash with ThinkingLevel.NONE skips internal chain-of-thought reasoning. The model receives the prompt and generates a response immediately, with no deliberation overhead. It sacrifices philosophical depth for chat-appropriate speed.

Can everyone use it?

No. DPS_Gemini maintains an approved_players.txt file. Messages from players not on this list are silently ignored. This prevents abuse and keeps the bot's resources focused on those who should have access.

What is the message limit?

Five messages per player per two-minute window. When the limit is hit, the bot sends you a countdown message telling you exactly how many seconds remain before your next allowed message. Freddison himself is exempt from this limit.

Does the bot promote DPS in every message?

Explicitly not. The system prompt instructs the bot to only mention DPS when it is genuinely relevant, and never to be "overly patriotic." The bot is a service, not a recruitment tool. It has dignity.

What happens if the API fails?

The bot retries up to three times with exponential backoff (2s, 4s, 6s). Different error types receive different responses: quota exhaustion, safety filters, invalid keys, and recitation blocks all trigger specific, informative messages to the requesting player.
